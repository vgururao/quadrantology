## Research scaffolding

The test now supports an opt-in research pathway for gathering calibration data on questions in `calibrating` status.

**How it works:** On the intro screen, if the current question set contains any calibrating questions, the user sees a consent block: "This test run includes N questions undergoing calibration. These will not be used to compute your result." Two choices: opt in (responses logged anonymously) or skip (calibrating questions dropped entirely). The consent block does not appear if no questions are in `calibrating` status.

**What's recorded when opted in:** The full test run — all question/answer pairs, not just the calibrating ones. This enables cross-correlation across questions, which is richer than per-question aggregates. The outcome (archetype, E/V position, ethics simplex) is stored alongside so questions can eventually be evaluated for discriminative validity.

**Privacy design:** Two tokens are generated per opted-in session. `session_id` (a UUID) is the internal D1 primary key — it never leaves the server. `run_token` (a 32-char hex string) is stored in the user's v2 localStorage run record. If the user later submits their full logbook via `/api/research/submit-logbook`, the server can link the two records using `run_token` without creating a duplicate or exposing the internal key.

**Calibrating status in v2 records:** A new `calibrating: [qids]` field is added to every v2 run record, recording which questions were in `calibrating` status at run time. This is used by the logbook submission endpoint to correctly tag `is_calibrating` on each response row, even for runs submitted months later.

**New endpoints:**
- `GET /api/questions` — serves live+calibrating questions from D1, falls back gracefully to 503 if D1 unavailable
- `POST /api/research/record-response` — per-run opted-in submission; idempotent (INSERT OR IGNORE)
- `POST /api/research/submit-logbook` — bulk logbook submission; links existing sessions via `run_token`, creates new ones for runs without it

**D1 schema additions:** `questions`, `question_state_log`, `research_subjects`, `research_sessions`, `research_responses`, `admin_challenges` tables added to `worker/schema.sql`. `worker/seed-questions.sql` created with INSERT OR IGNORE statements for all 28 questions plus initial state log entries.

## Admin UI (`/admin/questions`)

A private question management page at `docs/admin/questions.html`, protected by ECDSA P-256 challenge-response authentication.

**Auth flow:** The admin generates a key pair in the browser (non-extractable, stored in IndexedDB). The public key (SPKI, base64url) is pasted into the `ADMIN_PUBLIC_KEYS` CF Pages env var. To log in: the page fetches a nonce from `/api/admin/challenge`, signs it with the private key, posts the signature to `/api/admin/auth`, receives an HMAC-signed session token (1-hour TTL) stored in sessionStorage. All subsequent admin API calls include this token; `/api/admin/_middleware.js` verifies it.

Using browser-native non-extractable keys means the private key cannot be exfiltrated — it exists only in the browser's key storage on the admin's machine.

**Question editor features:**
- Sidebar with All / Live / Calibrating / Archived filter tabs
- A/B answer text, 5-dimension weight grid (exit, voice, virtue, consequentialist, deontological), response weight float, status toggle (3-button), notes field
- Status changes require a reason note, which is appended to the question's `state_log`
- Research data panel: A/B response bars with percentage, progress toward `calibration.min_samples` threshold, archetype breakdown for discrimination analysis
- State log panel: full audit trail of status changes
- "Import from questions.json" button: fetches the static seed file, maps legacy `active`/`retired` statuses, POSTs all questions to D1

## Personal Circle (`circle.html`)

The Personal Circle page is now built and functional as a client-side-only feature.

8 slots in a 4-column grid (2-col tablet, 1-col mobile). Empty slots have a dashed border and click to open an add modal. Filled slots show: contact name, archetype badge (exit teal / voice red), mini arc chips (oldest → newest with arrows, colored by group), E/V position bar with dot marker, slot age, and Update/Drop actions.

Adding a contact: paste a share URL into the modal. The page parses the `#v1:` fragment, tries `DecompressionStream('deflate-raw')` first, falls back to plain base64url JSON. The parsed contact is written to `quadrantology_circle` in localStorage.

**Demo mode:** A "Load demo data" button populates 3 slots with clearly sci-fi-named dummy contacts (Zev Okafor, Praxis Endolin, Sable Vorth) to let users explore the interface before they have real circle data. Demo slots are visually flagged with a small amber `demo` tag and slight opacity reduction. The demo bar message instructs users to drop the dummies to add real contacts. The 5 remaining empty slots are visible alongside the 3 demo contacts.

## Relationship Analysis (`analysis.html`)

The Relationship Analysis page computes views over circle + own data. No new data is collected or stored — purely a computed read over existing localStorage.

**Selection:** Left sidebar lists yourself (from most recent v2 run) and all filled circle slots as checkboxes. The Analyze button enables once 2+ are selected.

**Four analysis types:**

1. **Working Dynamic** — Exit/Voice composition bars per person, group narrative (balanced / Exit-heavy / Voice-heavy with threshold logic), archetype pair interaction notes for 9 specific E/V combos (e.g. "Hackers disrupt; Operators maintain. The Operator will feel like their work keeps getting undermined.")

2. **Values Map** — Group average ethics simplex bars, narrative keyed to dominant orientation (virtue / consequentialist / deontological), individual profiles, outlier callout when someone's ethics vector diverges > 0.18 from group average.

3. **Fault Lines** — All pairwise combinations ranked by largest single-dimension gap (E/V + 3 ethics dimensions). Top-pair narrative explains the specific tension. Table shows up to 5 pairs with bar visualisation.

4. **Collective Blind Spots** — Flags E/V split below 25% and ethics orientations below 22% average; narrative explains why underweight frames deserve explicit attention before major decisions.

**Demo mode:** Same 3 sci-fi contacts as circle.html plus a "You" / Holy Warrior own run. Demo bar links to circle.html to drop dummies once real contacts are available.

## Privacy discussion and roadmap additions

User asked whether the research scheme is provably private. Honest answer: no web app is provably private, because the server serves the code and secret telemetry would be undetectable without network auditing. Three mitigations added to the roadmap:

- **Tier 1:** Payload preview modal — show the user the exact JSON being sent before any research POST fires. Not a summary — the raw object, so they can verify name/notes/code are absent.
- **Tier 1.5:** Offline-verifiable bundle — signed zip of `docs/` with a SHA256 in a signed manifest. Users can download, verify the hash, and run locally with `python3 -m http.server`. Research logging doesn't fire in this mode unless the user explicitly initiates it.
- **Someday/Maybe:** ZK-verifiable local client — content-addressed signed bundle, test runs entirely local, research submission uses a ZK commitment scheme (selective opening) so the user can prove correct computation without revealing withheld data. Requires ZK-executable scoring function (RISC Zero / zkWASM) and a commitment scheme for the Q/A vector. Offline bundle is the near-term stepping stone.
