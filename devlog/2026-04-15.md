# Dev Log — April 15, 2026

**Session focus:** Moving from a static GitHub Pages site to a proper product — payments, access codes, persistent history, and a v2 data model.

---

## Starting point

Quadrantology had been a clean, dependency-free static site on GitHub Pages: six archetype pages, a theory page, a shop stub, and a 28-question personality test with client-side scoring. No backend, no payments, no persistence beyond the browser session. The test computed your archetype and showed scores, then you navigated away and it was gone.

The goal for this session was to build the minimal infrastructure needed to turn this into a real product — one that could handle payments, track your results over time, and scale toward richer features without over-engineering the foundation.

---

## Platform migration: GitHub to Cloudflare

The first decision was the hosting stack. The requirements were Stripe payments, per-user result persistence, and a future path to Ethereum/x402 payments. Cloudflare Pages + Workers was the clear fit: it deploys from the same GitHub repo with no build step (matching the site's no-framework philosophy), Workers run on the same domain (important for x402), and Cloudflare has a first-party x402 facilitator for when we get there.

The migration itself was trivial — point CF Pages at the `docs/` folder, push to master, done. The site went live at `quadrantology.pages.dev` in about 30 seconds. A DNS cutover to `quadrantology.com` is pending until the full payment stack is tested. We tagged `ghpages-v1` as a rollback point and documented the restore procedure in `CLAUDE.md`.

## UX rework: name, history, paywall

Before touching any backend, we rebuilt the test's UX flow from scratch.

The test previously just started at Q1. Now it opens with a name screen. Returning users see "Welcome back, [name] — you've taken this test N times" with a prominent Begin button. New users see their name input alongside a plain-language notice about the paywall they'll hit after completion.

Results are now saved to `localStorage` as a history array, with full export to JSON and re-import on another device. The download dialog warns that the file contains the user's personal access code. A history viewer page (`history.html`) shows all past runs. The "clear history" button triggers a two-step modal — "Download your history first?" with "Yes, download" and "I already did" options — because we never want to delete local data without giving the user a chance to save it first.

The paywall itself appears between the last question and the results. If a valid code is in localStorage, it's consumed silently and results appear. If not, the user sees two options: enter an access code, or buy access.

## Access code system

The core concept: every form of access is a code with a use counter in Cloudflare D1. Three types:

- **`personal`** — generated by the Stripe webhook on payment. Stored silently in localStorage. The user never sees or types it.
- **`coupon`** — batch-generated via an admin API endpoint. Human-readable (`QNTLG-XXXX-XXXX-XXXX`). Given out for promos, review copies, etc.
- **`org`** — same as coupon but tagged with an org identifier. Orgs buy a block of codes; their members enter them individually.

A key design principle: **we sell access, not data**. Org codes give employees access to the test. What those employees do with their results is entirely their own business. We deliberately do not build any mechanism for orgs to collect or compel sharing of employee results. If a team wants to compare results, each person can choose to share their own result via a URL — that's Tier 1 on the roadmap.

The Stripe integration uses Checkout (hosted page, redirect flow) with a Cloudflare Worker webhook handler that verifies the HMAC-SHA256 signature without needing the Stripe SDK. On successful payment, the webhook generates a personal code and writes it to D1. The client polls `GET /api/session-code?session_id=...` on return from Stripe, with a few retries to handle webhook processing lag.

Price is fetched live from Stripe at page load (`GET /api/price`) so the amount shown always matches the product — no hardcoded values in the frontend.

End-to-end test-mode verification: take the test, hit paywall, use card `4242 4242 4242 4242`, get redirected back, results appear. Code in localStorage, included in downloaded JSON. ✅

## v2 data model

The original history record was lossy: it stored computed scores and a flat answers array, but not which questions were asked or what their weights were at the time. This would become a problem once we add question sampling (showing different subsets on different runs) and as we evolve question weights.

The v2 record stores:

- `run`: an ordered array of `{qid, ans}` pairs — the exact questions shown and answers given
- `scores`: named object (exit, voice, virtue, consequentialist, deontological)
- `position`: continuous coordinates — `ev` (-1 to +1, Voice to Exit) and an ethics simplex (virtue/consequentialist/deontological proportions summing to 1.0)
- `questions_version`: which version of the question bank was used

The continuous position is the key addition. Rather than just a discrete archetype label, every result is a point in a 2D space: the E↔V axis and a 3-way ethics simplex. This enables interpolation between archetypes, trend visualization, and eventually the circle comparison feature where you can see where a group of people cluster relative to each other.

All 28 questions got permanent IDs (Q001–Q028) and a `weight` field (default 1.0). Questions are never deleted from the JSON — only retired — so old history records can always be replayed. Two questions (Q011 and Q023) have one all-zero weight answer; flagged for the author to review.

The history viewer was updated to normalize v1 and v2 records into a common shape for display, so old records continue to work alongside new ones.

## Roadmap shaped

The session ended with a roadmap planning discussion. The next major feature is **encrypted URL fragment sharing** — the same pattern used by CryptoPad and Excalidraw, where result data is compressed into a URL fragment that the browser never sends to the server. This becomes the foundation for a circle comparison tool (`compare.html`) where you paste in multiple people's result URLs and get a relational analysis — complementary archetypes, likely friction points, group dynamics. Fully client-side, no private data stored server-side.

The broader priority stack:

1. Encrypted URL sharing + circle comparison
2. Mobile-friendly UX pass
3. Email reminders at the 3-month and 5.5-month marks (Stripe captured the buyer's email)
4. Improved test with question sampling from an expanded bank
5. History analytics (trend charts, drift over runs)
6. Bulk org coupon admin UI
7. LLM results interpretation (in-page chat via Claude API)
8. x402/Ethereum payments
9. Google Drive sync
