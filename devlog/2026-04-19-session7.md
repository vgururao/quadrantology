### Touch active-state bug ‚Äî three failed attempts, one redesign

**The bug:** on iPad (Safari and Chrome), tapping a choice button left it visually highlighted on the next question. The reused DOM node carried its CSS `:active` state forward.

**Attempt 1:** `blur()` on both buttons ‚Äî removes `:focus` but not `:active`.

**Attempt 2:** `disabled = true` ‚Üí update text ‚Üí `disabled = false` synchronously ‚Äî browser batches the two state changes in the same JS task, never paints the intermediate disabled state, `:active` persists.

**Attempt 3:** `disabled = true` ‚Üí update text ‚Üí `disabled = false` in a `requestAnimationFrame` callback ‚Äî iOS Safari doesn't guarantee a visual paint between the JS task end and the rAF callback; still didn't clear `:active`.

**Attempt 4:** Remove `.choice-btn:active` CSS entirely ‚Äî `:active` still applied (confirmed by user on 1.0.3), so something else was persisting; or the state was genuine `:active` from a different source.

**Root fix:** Select-then-confirm UX (user's suggestion). Choice buttons now *select* rather than immediately advance. A separate Next ‚Üí button advances to the next question. Since navigation is a separate tap on a fresh button, the `:active` state never carries forward between questions. JS-controlled `.selected` class replaces any CSS pseudo-state for showing the chosen answer.

### Select-then-confirm question flow

Applied identically to both `gh-pages` (v1.1.0 ‚Üí v1.1.1) and `master` (v2.1.0-dev).

**HTML:** Added `<p class="response-changed" id="response-changed">` between choice buttons and nav. Nav row is `<div class="question-nav">` with Back and Next always rendered (no `display:none` toggling).

**JS:**
- `selectAnswer(choice)` ‚Äî records tentative answer, toggles `.selected` class, enables Next. If a prior different answer existed, shows "Response changed!" for 2 seconds.
- `advance()` ‚Äî commits current answer, increments `current`, calls `showQuestion()` or `computeResults()`.
- `showQuestion()` ‚Äî clears `.selected` on both buttons, restores it if question was previously answered (back-nav), sets `disabled` state on Back (Q1) and Next (no answer yet), clears any lingering message.
- `goBack()` ‚Äî decrements `current`, calls `showQuestion()` (restores previous selection).

**CSS:**
- `.choice-btn.selected` ‚Äî accent border + light background + bold weight
- `.choice-btn:active` ‚Äî restored (safe now; tapping a choice no longer causes page transition)
- `.question-nav` ‚Äî flexbox row, space-between
- `.next-btn` ‚Äî accent-coloured filled button
- `.back-btn` / `.next-btn` disabled states ‚Äî `opacity: 0.3`, `cursor: default`
- `.response-changed` ‚Äî `min-height: 1.3em` (no layout shift), accent colour, bold

**Side benefit:** users can now revise any answer before final submission by pressing Back, re-selecting, and pressing Next again.
