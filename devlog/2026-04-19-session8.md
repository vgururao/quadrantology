# Session 8 — 2026-04-19

## What was built

### Sampler rewrite: pairwise tie-free + co-occurrence-aware

`functions/api/sample-questions.js` was fully rewritten around two new ideas:

**Pairwise tie-free guarantees.** Questions are classified into four comparison types (ev, vc, vd, cd) based on their weight vectors. The sampler enforces odd counts for each type — making pairwise ties structurally impossible on all four axes. Circular orderings (virtue > conseq > deont > virtue) are valid results under this model. Targets live in `protocol.json` and default to ev:5, vc:3, vd:3, cd:3.

**Co-occurrence-aware selection.** The sampler now loads the last 200 rows from `question_sequences` in parallel with the questions query, builds a pair-frequency map, and penalizes candidates whose co-occurrence with already-selected questions is high. This prevents the same combinations from locking in across the user population — important when the pool is small (e.g. only 4 vc-type questions, need 3 per run).

The sort key is a 5-tuple: [types still needed desc, co-occurrence penalty, multi-run freshness, days since last sampled desc, times sampled asc]. This is evaluated greedily; the candidate pool is re-sorted on each pick so co-occurrence penalties stay current as questions are selected.

Multi-run overlap avoidance was redesigned: `test.html` now sends `previous_runs` (last 3 runs' QID arrays, newest first) instead of a single `previous_qids` array. The sampler builds `prevSet` (union of all, soft penalty) and `lastRunSet` (most recent only, stricter calibration-slot constraint) from this.

Fire-and-forget side effects: `times_sampled` + `last_sampled_at` incremented on each draw; sequence logged to `question_sequences`.

### D1 schema additions

- `question_sequences` table — compact sequence log, one row per draw
- `times_sampled` + `last_sampled_at` columns on `questions`
- `draft` added to the questions status CHECK constraint (new default for freshly created questions)

The `draft` lifecycle: questions are created in `draft` status by default (not served), promoted to `calibrating` via the admin UI to enter the pool and start collecting data, then to `live` once calibrated.

### Diagnostic infrastructure

`functions/api/admin/sequences.js` — new protected endpoint, GET with params `window_days`, `recent`, `include`. Returns summary stats, type balance (min/max/avg/p50 per comparison axis), per-question exposure frequencies, top-50 co-occurring pairs, recent sequence rows.

`docs/admin/diagnostics.html` — admin page for the above. Same ECDSA auth flow as `questions.html`. Shows summary stat cards (parity fail rate color-coded), type balance grid, exposure table with bar chart, co-occurrence table, recent sequences with parity badge.

`docs/admin/questions.html` topbar updated with Questions | Diagnostics nav.

### Smoke test script

`scripts/test-sampler.sh` — runs N sequential sample calls chaining each run's QIDs as `previous_runs` into the next call. Prints per-run: question count, parity, type counts vs targets, overlap count/pct, cooccurrence window, and full question text (truncated). Accepts optional base URL for testing against local wrangler dev.

## Key observations from test run

Running 4 draws in sequence against the live API showed:
- All draws: 13 questions, parity OK, type counts exactly at targets
- `cooc window 0` on all draws — the sequence log was empty before these runs; co-occurrence penalties will activate after real usage accumulates
- Q006, Q008, Q011, Q014, Q020 appeared in 3–4 of the 4 draws despite overlap avoidance — expected, because with 28 live questions and a 14-question target, some questions are structural linchpins (Q006 is the only dual-coded ev+vc question). Adding more pool diversity is the fix, not sampler tuning.

## Key decisions

**`previous_runs` window = 3.** Enough for meaningful overlap avoidance without over-constraining the sampler against a small pool. With 28 live questions and a 14-question target, going beyond 3 would start excluding more than half the pool as "recently seen."

**Admin keys are per-device, per-browser.** The ECDSA keypair lives in IndexedDB and is non-extractable. Multiple keys can be added to `ADMIN_PUBLIC_KEYS` (comma-separated, no spaces) — one per device/browser. Revoking a device means removing its key from the env var and redeploying.
