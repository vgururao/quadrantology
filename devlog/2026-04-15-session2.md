## GitHub Pages deploy fix

GitHub Pages had been silently auto-deploying from `master` ever since the Cloudflare migration began. Every new feature — the name intro screen, paywall, Stripe integration, v2 data model — had gone live at `quadrantology.com` automatically, even though the CF stack was still in test mode on a staging domain. The `ghpages-v1` tag was a bookmark, not an actual deploy gate.

Fixed by creating a `gh-pages` branch pinned to the `ghpages-v1` commit (pre-paywall, pre-name-intro), reconfiguring GitHub Pages to serve from that branch, and pushing a trivial change to force a CDN flush. Going forward: `master` pushes deploy to Cloudflare Pages only; `gh-pages` is frozen and only updated intentionally. The two deploy targets are now actually isolated.

## Reframe: personality tracker, not personality test

The most significant conceptual shift of the session. The product is not a test — tests have single authoritative results. This is a tracker, like a CGM or sleep tracker, where the trend over time is the product, not the spot reading.

This reframe drove several concrete changes:

**Journal entry on results.** After completing a run, users now see a freeform text box: *"Note recent or upcoming life events that might help interpret this result (private to you)."* The primary action button became **"Add results and journal entry to logbook"** — the save and the download are one gesture, and the artifact is now called a *logbook*, not a history. The note is saved to the v2 record (`note` field) and rendered in the logbook view. It is never included in share payloads.

**Logbook copy throughout.** "History" replaced with "logbook" across `test.html` and `history.html`. Download filename changed to `quadrantology-logbook.json`. The clear button and empty state copy updated to match.

**Design document.** `DESIGN.md` created at the repo root with founding principles:
1. Personality tracker, not personality test
2. CGM/sleep tracker model — trend monitoring, not spot tests (with Dexa scan as a complementary reference for the one-off vs. subscription portfolio model)
3. Personal Circle — intimate contacts, not a social network; bilateral friction-ful sharing by design
4. Data model first — all feature design starts by updating `DATAMODEL.md`

## Data model and protocol spec

`DATAMODEL.md` created as the canonical data model document. The app is stateless — no server-side user data — so the logbook *is* the product, and all feature design should flow from the data model.

The full data model now covers four structures:

1. **Own run records** (v1 legacy + v2 current) — unchanged except the new `note` field
2. **Personal Circle entries** (`circle-v1`) — point-in-time arc snapshots from others, up to `max_slots` entries
3. **Share URL payload** — generated on demand, name set at share time (not persisted), most recent 3 runs in summary form only, no Q/A or notes
4. **Coach Mode client logbooks** (`client-v1`) — full imported logbooks, Coach Mode subscription only, encrypted before transfer

`docs/data/protocol.json` created to hold all feature thresholds and limits as runtime-readable parameters rather than hardcoded values: `min_runs`, `window_days`, `ics_interval_days`, `sharing.min_runs_to_share`, `personal_circle.max_slots`, etc. Pages should read this file rather than hardcode any of these.

## Product portfolio specced

Three tiers, shaped by the Dexa scan model (one-off scan vs. annual membership):

**Single assessment sequence** — one-off purchase. 3 tests over 90 days. Basic logbook and trendline analytics on completion.

**Annual subscription** — ongoing tracking, renews yearly. Unlocks Personal Circle (up to 8 slots), Relationship Analysis (standard mode using summary arcs), richer analytics tiers, .ics retake schedule, and eventually LLM chat.

**Coach Mode** — separate subscription for practitioners. Unlocks full client logbook import (encrypted file transfer — asymmetric key pair generated in-browser, private key never leaves coach's device), and deep relationship analysis across full logbooks rather than summary arcs. Late roadmap; build when there's enough traction to warrant a coaching use case.

## Features specced (stub level)

**URL fragment sharing.** "Share my arc" button requires `min_runs_to_share` completed runs. User enters a nickname at share time (not persisted, can differ per recipient). Payload: name, most recent 3 archetypes + summary scores + datestamps — no Q/A data, no journal notes. ~300 bytes raw, ~220 chars in fragment after compression. `r.html` renders received arcs client-side; fragment never sent to server.

**Personal Circle.** Up to 8 slots (`personal_circle.max_slots`) for intimate contacts. Not a social network — no discovery, no feed. Each slot: arc mini-view, Update (paste new URL), Drop. `r.html` shows "Add to Personal Circle" — active for subscribers, inactive with "requires subscription" note for others. Stored in `quadrantology_circle` localStorage key, included in downloaded logbook.

**Relationship Analysis page** (`analysis.html`). Select 2+ people from Personal Circle + optionally yourself, choose an analysis type, get a computed result. Yourself is optional (mediator use case: analyze two friends' dynamic on their behalf). Standard mode uses summary arcs; deep mode (Coach Mode) uses full logbooks. Pure computed view — no new data stored.

**Analytics page.** Rename of `history.html` → `analytics.html` planned. Locked until `min_runs` runs completed. Trendline chart + commentary. Gets richer at 5 and 10 runs. Parameters in `protocol.json`.

**.ics retake schedule.** Client-side generation of a calendar file with retake events spaced ~30 days apart. Anchor: first test date. No server involved, no email required — fully anonymous, works for crypto-paying users.
